# 3.2. Опис головних функцій програми

Функціональність програми розподілена між сімома модулями, кожен з яких відповідає за визначений аспект роботи застосунку: модель даних, алгоритми маршрутизації, низькорівневий рендер, побудова графа, бібліотека заготовлених топологій, візуалізація алгоритмів та користувацький інтерфейс. Далі наведено опис головних функцій кожного модуля з їхніми сигнатурами та коротким поясненням призначення.

## 3.2.1. Модуль `NetworkGraph.h`

Модуль містить клас `NetworkGraph`, який реалізує неорієнтований зважений граф мережі маршрутизаторів. Для побудови та доступу до графа використовуються чотири головні методи.

Метод `void addNode(int id, const string& name, int x = 0, int y = 0)` додає у граф нову вершину із заданим ідентифікатором, ім'ям та координатами на 2D-полі. Якщо для вершини ще не існує запису у списку суміжності, метод створює порожній вектор ребер. Координати `x` та `y` одночасно використовуються як логічна позиція для розміщення вершини у графі та як екранна позиція для рендера, що спрощує логіку візуалізації.

Метод `void addEdge(int u, int v, double weight, bool isDirected = false)` додає ребро між двома вершинами з заданою вагою. Параметр `isDirected` за замовчуванням набуває значення `false`, тобто стандартна поведінка полягає у додаванні двох дзеркальних ребер — від `u` до `v` та у зворотному напрямку з однаковою вагою. Таким чином, за замовчуванням граф є неорієнтованим, що відповідає фізичній природі каналів зв'язку у комп'ютерних мережах.

Метод `const vector<Edge>& getNeighbors(int id) const` повертає список вихідних ребер для заданої вершини, що використовується усіма алгоритмами маршрутизації для обходу сусідів поточної вершини. Метод `const Node& getNode(int id) const` повертає дані про окрему вершину за її ідентифікатором, а метод `const unordered_map<int, Node>& getNodes() const` — всі вершини графа у вигляді хеш-таблиці, що необхідно для ітерації по всіх вершинах під час рендера.

## 3.2.2. Модуль `RoutingAlgorithms.h`

Модуль містить клас `RoutingAlgorithms`, який реалізує три статичні методи пошуку найкоротшого шляху з єдиною уніфікованою сигнатурою. Така уніфікація дозволяє викликати будь-який з алгоритмів через один і той самий вказівник на функцію, що використовується у модулі візуалізації для послідовного прогону трьох алгоритмів.

Метод `static RoutingResult dijkstra(const NetworkGraph& graph, int startId, int targetId, StepLog* log = nullptr)` реалізує алгоритм Дейкстри на основі пріоритетної черги (priority queue). Функція ініціалізує таблицю відстаней значенням нескінченності, послідовно вилучає з черги вершину з найменшим поточним значенням відстані, релаксує всі її вихідні ребра та записує кожен крок у журнал подій, якщо вказівник `log` не дорівнює `nullptr`. Детальна блок-схема алгоритму зображена на рисунку 2.7.

Метод `static RoutingResult bellmanFord(const NetworkGraph& graph, int startId, int targetId, StepLog* log = nullptr)` реалізує алгоритм Беллмана-Форда, який виконує `V − 1` проходів по всіх ребрах графа з релаксацією значень відстаней на кожному проході. Алгоритм коректно обробляє графи з від'ємними вагами, однак має вищу асимптотичну складність. Блок-схема алгоритму наведена на рисунку 2.8.

Метод `static RoutingResult aStar(const NetworkGraph& graph, int startId, int targetId, StepLog* log = nullptr)` реалізує алгоритм A\* з евклідовою евристикою. На відміну від Дейкстри, у пріоритетну чергу вкладається значення `f = g + h`, де `g` — вже обчислена відстань від стартової вершини, а `h` — евристична оцінка, рівна евклідовій відстані від поточної вершини до цільової, обчислена за формулою `sqrt((x1 − x2)² + (y1 − y2)²)`. Блок-схема алгоритму наведена на рисунку 2.9.

Допоміжна статична функція `static double euclideanHeuristic(const Node& a, const Node& b)` обчислює евристику за координатами вершин і використовується методом `aStar` при кожному оновленні пріоритету у черзі.

## 3.2.3. Модуль `ConsoleRenderer.h`

Модуль реалізовано у вигляді простору імен `ConsoleRenderer`, оскільки всі функції є операціями без стану. Модуль інкапсулює низькорівневий рендер у фіксованій області 80×20 символів консолі.

Функція `inline void gotoXY(int x, int y)` є обгорткою над функцією `SetConsoleCursorPosition` із Windows Console API і переміщує курсор консолі у задану позицію. Функція `inline void setAttr(WORD attr)` встановлює поточний атрибут тексту (колір символу та фону) через `SetConsoleTextAttribute`. Ці дві функції є фундаментальними примітивами, на яких побудовано увесь рендер.

Функція `inline void drawNode(int x, int y, int id, WORD attr)` малює у заданій позиції вершину графа у вигляді рядка `"[id]"` заданим кольором. Перед виводом координата `screenX` обмежується розміром поля, щоб мітка не виходила за праву межу.

Функція `inline void drawLine(int x1, int y1, int x2, int y2, WORD attr)` малює ребро графа між двома точками псевдографічними символами `-`, `|`, `\` або `/`, вибір яких залежить від кута нахилу ребра. Для інтерполяції використовується класичний підхід цифрового диференціального аналізатора з параметром `t = i / steps`, а перший і останній символи ребра пропускаються, щоб лінія не перекривала мітки вершин.

Функція `inline void drawWeight(int x1, int y1, int x2, int y2, double weight, WORD attr)` виводить число, що відповідає вазі ребра, поблизу середини ребра. Значення ваги при цьому приводиться до цілочисельного типу для компактнішого відображення.

Функція `inline void drawField(const NetworkGraph& graph)` виконує повне перемальовування поля і є основним входом рендера. Послідовність дій є фіксованою: спочатку очищається поле пробілами, потім малюються всі ребра та їх ваги, після чого поверх ребер малюються всі вершини. Для уникнення подвійного малювання неорієнтованих ребер використовується допоміжна множина `drawnEdges` з канонічним ключем у форматі `(min, max)`.

## 3.2.4. Модуль `GraphBuilder.h`

Модуль `GraphBuilder` реалізує інтерактивну побудову графа користувачем та складається з трьох головних функцій.

Функція `inline NetworkGraph placeNodesInteractive()` забезпечує першу фазу ручної побудови — інтерактивне розміщення вершин на 2D-полі. У циклі на кожній ітерації перемальовується поле з поточним станом графа та символом курсора `+` у поточній позиції. Користувач переміщує курсор клавішами-стрілками, додає вершину у поточну позицію клавішею Enter та завершує фазу клавішею Esc. Наступний ідентифікатор вершини автоматично збільшується після кожного додавання.

Функція `inline void addEdgesByText(NetworkGraph& g)` реалізує другу фазу — текстове введення ребер. Користувач у форматі `<id_from> <id_to> <weight>` послідовно вводить рядки, кожен з яких описує одне ребро. Функція виконує валідацію: перевіряє коректність формату та наявність обох вершин у графі, виводячи відповідні повідомлення про помилки червоним кольором. Введення завершується порожнім рядком.

Функція `inline NetworkGraph buildInteractive()` об'єднує дві попередні фази в єдиний сценарій ручної побудови графа: спочатку викликає `placeNodesInteractive`, потім, за умови що користувач додав хоча б одну вершину, викликає `addEdgesByText`. Блок-схема всього сценарію ручної побудови графа наведена на рисунку 2.6.

## 3.2.5. Модуль `TopologyLibrary.h`

Модуль `TopologyLibrary` містить три функції-фабрики, що повертають готові графи зі заздалегідь визначеними топологіями. Координати вершин у всіх трьох топологіях підібрано вручну так, щоб у полі 80×20 граф виглядав симетрично і ребра не перекривали вершин.

Функція `inline NetworkGraph makeStar()` створює топологію «зірка» з шести вершин: центральна вершина `R1` у позиції `(40, 10)` з'єднана ребрами з п'ятьма периферійними вершинами `R2`–`R6`. Ваги ребер варіюються від 4 до 8, що моделює канали різної затримки. У такій топології будь-який шлях між двома периферійними вершинами проходить через центральну.

Функція `inline NetworkGraph makeMesh()` створює щільну топологію «сітка» з п'яти вершин та восьми ребер. Існує кілька альтернативних маршрутів між однією і тією ж парою вершин з різними сумарними вагами, що дозволяє продемонструвати відмінності у поведінці алгоритмів — зокрема перевагу A\* завдяки евристичному відсіванню неоптимальних шляхів.

Функція `inline NetworkGraph makeLinear()` створює топологію «ланцюг» з шести послідовно з'єднаних вершин `R1`–`R6`. Єдиний можливий шлях між будь-якими двома вершинами проходить через усі проміжні, що робить цю топологію корисною для демонстрації однакового шляху, знайденого усіма трьома алгоритмами, при різних значеннях метрик виконання.

## 3.2.6. Модуль `AlgorithmVisualizer.h`

Модуль `AlgorithmVisualizer` забезпечує покрокову візуалізацію роботи алгоритмів, анімацію передачі пакета та формування порівняльного звіту. Модуль базується на абстракції — журналі подій `StepLog`, який заповнюється алгоритмом під час обчислення та покроково відтворюється візуалізатором з налаштовуваною затримкою.

Функція `inline void runAllAlgorithms(const NetworkGraph& g, int startId, int targetId)` реалізує головний сценарій — послідовний прогін трьох алгоритмів на одному графі. Функція формує масив вказівників на функції `dijkstra`, `bellmanFord` та `aStar`, для кожного з них викликає функцію `runSingle`, зберігає отриманий результат у масив, а після завершення всіх трьох прогонів викликає функцію `renderComparisonReport` для побудови порівняльної таблиці.

Функція `inline RoutingResult runSingle(const string& algoName, AlgoFn fn, const NetworkGraph& g, int startId, int targetId, SpeedState& ss)` виконує прогін одного алгоритму з візуалізацією. Спочатку алгоритм запускається з передачею журналу подій. Після завершення обчислення функція послідовно проходить по записах журналу, виводячи на кожному кроці поточний стан графа через функцію `repaintStep` та інформаційну панель через функцію `drawInfoPanel`. Після відтворення всіх кроків викликається функція `highlightPath` для підсвічування фінального шляху, а потім `animatePacket` для анімації передачі пакета обраним маршрутом.

Функція `inline void repaintStep(const NetworkGraph& g, const StepEvent& ev, int startId, int targetId)` перемальовує вершини графа у кольорах, що відповідають поточному стану алгоритму: стартова вершина — синім, цільова — червоним, відвідані — темно-зеленим, фронтир черги — пурпуровим, поточна вершина — жовтим. Це дозволяє користувачеві наочно спостерігати за поширенням фронту пошуку.

Функція `inline void highlightPath(const NetworkGraph& g, const vector<int>& path, int startId, int targetId)` малює фінальний шлях яскраво-зеленим кольором, підсвічуючи всі ребра та вершини, що входять у знайдений маршрут.

Функція `inline void animatePacket(const NetworkGraph& g, const vector<int>& path, SpeedState& ss, const string& algoName)` реалізує анімацію передачі пакета від стартової до цільової вершини. Символ `*` послідовно рухається уздовж кожного ребра шляху, що ілюструє маршрутизацію у реальній мережі.

Функція `inline void renderComparisonReport(const string algoNames[3], const RoutingResult results[3], const NetworkGraph& g, int startId, int targetId)` будує порівняльну таблицю метрик трьох алгоритмів. Для кожної метрики (довжина шляху, вартість, кількість ітерацій, релаксацій, розкритих вершин, час виконання) формується рядок таблиці, в якому найкраще та найгірше значення виділяються відповідно зеленим та червоним кольорами. Після таблиці виводиться текстовий висновок із переможцями за кожною метрикою.

Допоміжна функція `inline void pollInput(SpeedState& ss)` реалізує неблокуюче зчитування клавіатури: перевіряє наявність натиснутої клавіші через `_kbhit`, обробляє клавішу Space як паузу, стрілки вгору та вниз як зміну швидкості, Esc як пропуск поточного алгоритму. Функція `inline void sleepResponsive(int ms, ...)` забезпечує «розумну затримку» між кроками візуалізації, яка періодично перевіряє стан керування через `pollInput`, що дозволяє миттєво реагувати на натискання клавіш без очікування завершення повної затримки.

## 3.2.7. Модуль `kursova.cpp`

Файл `kursova.cpp` є єдиною одиницею трансляції програми та містить точку входу, головне меню та допоміжні екрани інтерфейсу.

Функція `int main()` — точка входу програми. Функція виконує налаштування кодової сторінки консолі у режим CP1251 для коректного відображення кириличних символів, ховає курсор консолі та запускає головний цикл меню. У циклі на кожній ітерації очищується екран, перемальовується меню з чотирма пунктами «Play», «Help», «About» та «Exit», активний пункт виділяється інверсією кольорів, після чого зчитується клавіша користувача через `_getch` та виконується перехід до відповідної функції або завершення програми.

Функція `void showPlay()` виводить підменю симуляції з трьома варіантами: побудова графа вручну, вибір заготовленої топології або повернення у головне меню. При виборі першого варіанту викликається функція `GraphBuilder::buildInteractive`, а результат передається у функцію `finalizeGraph`. При виборі другого варіанту викликається функція `showTopologySelection`.

Функція `void showTopologySelection()` реалізує підменю вибору заготовленої топології — Star, Mesh або Linear. При виборі конкретної топології викликається відповідна функція з модуля `TopologyLibrary`, а отриманий граф передається у функцію `finalizeGraph`.

Функція `void finalizeGraph(const NetworkGraph& g, const string& title)` є єдиною точкою переходу від фази побудови графа до фази візуалізації. Функція викликає `renderGraphScreen` для показу готового графа з заголовком, а потім `selectEndpointsAndPreview` для вибору стартової та цільової вершин та запуску прогону алгоритмів.

Функція `void selectEndpointsAndPreview(const NetworkGraph& g)` запитує у користувача ідентифікатори стартової та цільової вершин, виконує валідацію введених даних (перевіряє їх існування у графі та неспівпадіння) і, у разі успіху, викликає `AlgorithmVisualizer::runAllAlgorithms`. У разі помилки виводиться повідомлення червоним кольором.

Функції `void showHelp()` та `void showAbout()` виводять довідкову інформацію та відомості про програму відповідно. Функція `showHelp` описує клавіші керування у форматі «клавіша — дія», згрупованих за розділами «Меню», «Ручна побудова графа», «Вибір маршруту» та «Візуалізація алгоритмів». Функція `showAbout` виводить коротку анотацію мети програми, перелік її функціонала та використані технології.

Загалом усі функції програми побудовані навколо кількох архітектурних принципів: уніфікованої сигнатури для алгоритмів, що спрощує їхнє послідовне виконання; відокремлення обчислювальної логіки від рендера через журнал подій; та фіксованого розміру поля візуалізації, що дозволяє всім візуальним компонентам використовувати єдину систему координат.
