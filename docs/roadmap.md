# Roadmap

Послідовний план реалізації симулятора — від поточного стану до повної відповідності `mission.md`. Кожен пункт = одна фіча, маленький крок, який можна завершити і перевірити окремо.

Умовні позначки:

- Вузол = маршрутизатор, розміщений на 2D полі консолі за координатами `(x, y)`.
- Ребро = канал зв'язку між двома маршрутизаторами з ваговим коефіцієнтом (вартістю).

---

## Фаза 0. Фікс бази

Перед додаванням нових фіч привести поточний код до робочого стану, щоб не тягнути помилки далі.

### Пункт 0.1. Виправити кирилицю в `NetworkGraph::printGraph`

Літерали в `kursova/NetworkGraph.h` не в CP1251 → в консолі mojibake. Перекодувати рядки так, щоб вивід збігався зі стилем решти програми (`kursova/kursova.cpp`).

### Пункт 0.2. Прибрати захардкоджене демо з `showPlay`

Зараз `showPlay()` (`kursova/kursova.cpp:32-88`) одразу створює 5-вузлову топологію і запускає Дейкстру від 1 до 5. Винести цю логіку і замість неї показати підменю вибору сценарію побудови графа (поки що — заглушки для наступних пунктів).

## Фаза 1. Побудова графа

Мета: користувач має два шляхи отримати граф маршрутизаторів — побудувати вручну або обрати заготовлений.

### Пункт 1.1. Рендер графа на 2D полі консолі

Намалювати наявний `NetworkGraph` у виділеній області консолі, використовуючи координати `x, y` кожного `Node` (вони вже зберігаються, але ніде не рендеряться). Ребра малюються між вузлами. Це фундамент для всіх подальших візуалізацій.

### Пункт 1.2. Інтерактивне додавання вузлів на поле

Режим «ручна побудова», частина 1: на 2D полі з'являється курсор, який користувач рухає клавішами (стрілки + Enter). Enter на порожній клітинці — додати вузол у цю позицію. Назва вузла генерується автоматично (`R1`, `R2`, …).

### Пункт 1.3. Додавання ребер через текстовий ввід

Режим «ручна побудова», частина 2: після розміщення вузлів користувач вводить ребра текстом — пари `<id_from> <id_to> <weight>` — поки не закінчить. Граф оновлюється на 2D полі після кожного доданого ребра.

### Пункт 1.4. Бібліотека заготовлених топологій

Декілька якісних заготовлених варіантів графа (різна щільність, розмір, наявність циклів) — користувач обирає один зі списку. На виході — готовий `NetworkGraph`.

### Пункт 1.5. Вибір стартового і цільового вузлів

Після отримання графа (будь-яким шляхом) користувач обирає, між якими двома вузлами шукати шлях. Ці id передаються алгоритмам на вхід.

---

## Фаза 2. Алгоритми маршрутизації

Мета: мати всі три алгоритми, зазначені в назві роботи, з уніфікованою сигнатурою, щоб UI міг викликати їх однаково.

### Пункт 2.1. Алгоритм Беллмана-Форда

Додати статичний метод `RoutingAlgorithms::bellmanFord(graph, startId, targetId)` з повертанням `RoutingResult`. Класична реалізація з `|V|-1` проходами релаксації.

### Пункт 2.2. Алгоритм A\*

Додати статичний метод `RoutingAlgorithms::aStar(graph, startId, targetId)`. Евристика — евклідова відстань між вузлами за координатами `x, y` (вони вже є у структурі `Node`). Повертання — той самий `RoutingResult`.

### Пункт 2.3. Розширення `RoutingResult` для порівняння

Додати поля, які знадобляться у фінальному звіті порівняння (наприклад, кількість релаксацій / розкритих вузлів, час виконання). Узгодити між усіма трьома алгоритмами.

---

## Фаза 3. Візуалізація роботи алгоритмів

Мета: користувач бачить, *як саме* працює кожен алгоритм, і може відстежити принцип його роботи.

### Пункт 3.1. Покрокова візуалізація на 2D полі

Під час роботи алгоритму на 2D полі кольорами виділяються: поточний вузол, відвідані вузли, фронтир (вузли в черзі/на розгляді), невідвідані, фінальний шлях. Після завершення — підсвічений оптимальний шлях.

### Пункт 3.2. Анімація передачі пакетів уздовж шляху

Після того як алгоритм знайшов шлях, «пакет» анімовано проходить від стартового до цільового маршрутизатора ребро за ребром — видно процес передачі між серверами.

### Пункт 3.3. Динамічне керування швидкістю

Під час будь-якої анімації (як покрокової візуалізації алгоритму, так і руху пакета) гарячі клавіші змінюють швидкість у реальному часі: `+` / `-` (або `↑` / `↓`) регулюють затримку між кроками; пауза / продовження — окрема клавіша.

### Пункт 3.4. Послідовний прогін усіх трьох алгоритмів

На одному і тому ж графі з однаковою парою start/target послідовно запускається візуалізація Дейкстри → Беллмана-Форда → A\*. Між прогонами — коротка пауза/підказка для переходу далі.

---

## Фаза 4. Порівняльний аналіз

### Пункт 4.1. Аналітичний звіт по трьох алгоритмах

Після прогону всіх трьох алгоритмів — таблиця з метриками (довжина шляху, сумарна вартість, кількість ітерацій / розкритих вузлів, час) для кожного алгоритму поруч + короткий текстовий висновок, який саме алгоритм виявився ефективнішим на цьому конкретному графі та чому.

## Фаза 5. Поліровка

### Пункт 5.1. Уніфікація стилю виводу

Єдина кольорова палітра, заголовки секцій, рамки і відступи в усіх екранах (меню, побудова графа, візуалізація, звіт).

### Пункт 5.2. Оновлення `Help` та `About`

Довідка (`showHelp`) описує всі клавіші керування (меню + побудова графа + клавіші швидкості під час анімації). `About` узгоджений з фінальним функціоналом.

### Пункт 5.3. Чек-ліст відповідності `mission.md`

Пройтися по всіх критеріях готовності з `docs/mission.md` і переконатися, що кожен виконаний.

## Чек-ліст відповідності `mission.md`

Перевірено на момент завершення Фази 5. Кожна вимога — з посиланням на файл/функцію, де реалізована.

### Функціональні вимоги

- [x] **Побудова графа маршрутизаторів** — `kursova/GraphBuilder.h` (ручна: курсор на 2D-полі + текстовий ввід ребер) і `kursova/TopologyLibrary.h` (3 заготовки: Star, Mesh, Linear).
- [x] **Ваги ребер (вартість/затримка)** — `kursova/NetworkGraph.h` (`struct Edge { int to; double weight; }`); `addEdge(u, v, w)` неорієнтовано за замовчуванням.
- [x] **Алгоритм Дейкстри** — `RoutingAlgorithms::dijkstra` (бінарна купа, ранній вихід на цільовому вузлі).
- [x] **Алгоритм Беллмана-Форда** — `RoutingAlgorithms::bellmanFord` (|V|-1 релаксацій + контрольна перевірка негативних циклів).
- [x] **Алгоритм A\*** — `RoutingAlgorithms::aStar` з евклідовою евристикою `sqrt(dx²+dy²)` за `Node.x, Node.y`.
- [x] **Покрокова візуалізація роботи алгоритму** — `AlgorithmVisualizer::runSingle` (кольорове підсвічування: start/target/current/visited/frontier/path).
- [x] **Анімація передачі пакета** — `AlgorithmVisualizer::animatePacket` (символ `*` рухається вздовж ребер знайденого шляху).
- [x] **Керування швидкістю/паузою/пропуском** — `AlgorithmVisualizer::pollInput` (Space, Up/Down, Esc); неблокуючий `_kbhit` під час анімації.
- [x] **Послідовний прогін трьох алгоритмів на одному графі** — `AlgorithmVisualizer::runAllAlgorithms` (Дейкстра → Беллман-Форд → A*).
- [x] **Порівняльний звіт** — `AlgorithmVisualizer::renderComparisonReport`: ASCII-таблиця метрик (довжина, вартість, ітерації, релаксації, розкрито, час), кольорове підсвічування найкращих/найгірших значень, список переможців за метриками, одне-речення висновок.

### Нефункціональні вимоги

- [x] **Windows console з `<windows.h>`** — `kursova/kursova.cpp`, `kursova/ConsoleRenderer.h`, `kursova/GraphBuilder.h`, `kursova/AlgorithmVisualizer.h`.
- [x] **Кирилиця в консолі** — `SetConsoleCP(1251)` / `SetConsoleOutputCP(1251)` у `main()`; файли з кириличними літералами мають UTF-8 BOM (MSVC транскодує в CP1251 на етапі компіляції).
- [x] **Єдиний стиль екранів** — усі заголовки у форматі `=== НАЗВА ===` кольором 10; єдина `waitKey(prompt)` замість різних варіантів «Натисніть…»; помилки — червоним (12); активний пункт меню — інверсія.
