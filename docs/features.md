## 1. Точка входу та налаштування консолі

**Файл:** `kursova/kursova.cpp` — функція `main()` (рядки 310–369).

**Суть.** Програма запускається в звичайному Windows-консольному вікні, виводить кирилицю, не блимає системним курсором під час анімацій.

**Реалізація:**

- `SetConsoleCP(1251)` / `SetConsoleOutputCP(1251)` перемикають кодову сторінку консолі на CP1251. Усі українські літерали у вихідному коді компілюються MSVC-компілятором у CP1251, тому текст в консолі рендериться без mojibake.
- `hideCursor()` (рядки 20–26) через Win32 API (`GetStdHandle(STD_OUTPUT_HANDLE)` + `SetConsoleCursorInfo` з `bVisible = FALSE`) ховає мигаючий курсор — це критично для анімацій, інакше каретка весь час стрибала б по полю.
- `setColor(text, bg)` (рядки 29–33) — обгортка над `SetConsoleTextAttribute`, яка кодує пару «текст/фон» у 8-бітний атрибут. Використовується всюди як єдиний спосіб керувати кольорами.

**Чому так.** Win32 Console API — це вимога курсового завдання (див. `docs/mission.md`), тому кольорів через ANSI-escape-коди немає; кожна зміна кольору — окремий системний виклик.

## 2. Модель даних: граф мережі

**Файл:** `kursova/NetworkGraph.h` (header-only, весь клас inline).

**Що моделюється.** Неорієнтований зважений граф маршрутизаторів. Вузол — маршрутизатор з координатами на 2D-полі, ребро — канал зв'язку з ваговим коефіцієнтом (затримкою).

**Структури даних:**

```cpp
struct Edge { int to; double weight; };
struct Node { int id; string name; int x, y; };
```

- `Node.x, Node.y` — це **одночасно** логічні координати графа і екранні координати рендера: жодних окремих «логічних» і «екранних» координат немає. Це спрощує код: те, що алгоритм A\* використовує як евристику (`sqrt(dx² + dy²)`), — це ті самі координати, за якими вузол малюється.

**Зберігання:**

- `unordered_map<int, Node> nodes` — `id` → `Node`.
- `unordered_map<int, vector<Edge>> adjList` — adjacency list, `id` → список вихідних ребер.

**Ключова деталь — `addEdge(u, v, weight, isDirected = false)`** (рядки 34–39): за замовчуванням вставляє **два ребра** (u→v і v→u) з однаковою вагою, тобто граф логічно неорієнтований. Якщо треба орієнтований — передати `isDirected = true`, але в поточних сценаріях це ніде не використовується.

**Чому так.**

- `unordered_map` замість `vector` дає пошук по `id` за O(1) — алгоритми часто звертаються до сусідів конкретного вузла через `getNeighbors(id)`.
- `using namespace std;` на рівні файлу — свідоме спрощення для невеликого header-only модуля.
- Координати зберігаються в самому `Node`, а не в окремій `Positions`-таблиці, щоб A\* міг обчислити евристику без додаткового контексту.

## 3. Меню та навігація

**Файл:** `kursova/kursova.cpp` — `main()` (рядки 321–362) та підменю `showPlay()` / `showTopologySelection()` (рядки 128–234).

**Суть.** Усі вибори користувач робить стрілками + Enter, без введення номерів пунктів.

**Як це працює.**

1. Кожен екран меню — це `while (running)` з `system("cls")` на кожній ітерації і наступним перемальовуванням списку пунктів.
2. Активний пункт візуалізується **інверсією** кольорів: `setColor(0, 15)` (чорний текст на білому тлі), решта — стандартно.
3. Читання клавіш — `_getch()` з `<conio.h>`. Win32-консоль повертає стрілки як **послідовність із двох байт**: префікс `224` (0xE0), потім скан-код (`KEY_UP = 72`, `KEY_DOWN = 80`). Тому код читає перший байт, порівнює з `224`, і тільки тоді читає другий:

```cpp
int key = _getch();
if (key == 224) {
    key = _getch();
    if (key == KEY_UP) { ... }
}
else if (key == KEY_ENTER) { ... }
```

4. Циклічна навігація: після останнього пункту стрілка вниз переходить на перший, і навпаки (`if (activeIndex < 0) activeIndex = size - 1`).

**Чому так.** `_getch()` — неблокуюче зчитування без луни, що дозволяє реагувати на натискання миттєво, без очікування Enter. `system("cls")` простіше, ніж ручне очищення буфера консолі, і нормально виглядає на частоті оновлення меню (кілька разів на секунду, а не кадр за кадром).

**Структура меню:**

- Корінь (`main`): `Play → Help → About → Exit`.
- `Play` (`showPlay`): `Побудувати вручну → Обрати топологію → Назад`.
- `Обрати топологію` (`showTopologySelection`): `Star → Mesh → Linear → Назад`.

Всі три рівні використовують однакову схему рендера й обробки клавіш — патерн повторюється, але не винесений у функцію, бо дії на Enter у кожному меню різні і абстракція була б штучною.

## 4. Рендеринг графа на 2D-полі консолі

**Файл:** `kursova/ConsoleRenderer.h` — namespace `ConsoleRenderer`.

**Суть.** Граф малюється у фіксованій області консолі **80×20 символів**, починаючи з рядка 3. Вузли — як `[id]`, ребра — ASCII-лініями, ваги — числом посередині ребра.

**Константи області:** `FIELD_TOP = 3`, `FIELD_WIDTH = 80`, `FIELD_HEIGHT = 20` (рядки 13–15). Усі інші частини UI (заголовки, панель стану, підказки) будуються навколо цих меж.

### 4.1. `gotoXY(x, y)` — переміщення каретки

Обгортка над `SetConsoleCursorPosition`. Використовується **скрізь**, щоб малювати «на місці», без гортання. Це ключ до анімацій: замість скролу ми перемальовуємо конкретні клітинки.

### 4.2. `drawLine(x1, y1, x2, y2, attr)` — малювання ребра

Класичне «digital differential analyzer» через інтерполяцію параметром `t = i / steps`. Особливість — **вибір символу** залежно від кута нахилу:

```cpp
if (adx >= ady * 2) ch = '-';       // майже горизонтально
else if (ady >= adx * 2) ch = '|';  // майже вертикально
else if ((dx > 0) == (dy > 0)) ch = '\\';  // ↘ або ↖
else ch = '/';                             // ↗ або ↙
```

Ребро не виходить за межі поля (`if (x >= 0 && x < FIELD_WIDTH ...)`). Перший і останній символ пропускаються (`for (int i = 1; i < steps; ++i)`) — щоб лінія не перекривала мітку вузла.

### 4.3. `drawWeight(x1, y1, x2, y2, weight, attr)` — вага на ребрі

Вага виводиться біля середини ребра: для горизонтальних/похилих ребер — на рядок нижче, для вертикальних — на тому ж рядку з зсувом по X. Число завжди кастується до `int`, бо всі ваги в проєкті цілочисельні (попри тип `double`), і так вони виглядають компактніше.

### 4.4. `drawNode(x, y, id, attr)` — маршрутизатор

Вузол — це рядок `"[id]"`. Перед виводом код обмежує `screenX` так, щоб увесь `[id]` уміщався в поле. Тобто коли користувач поставив вузол впритул до правого краю, мітка зсувається всередину.

### 4.5. `drawField(graph)` — повний рендер

Порядок шарів критичний:

1. **Очистити поле** (`clearField()` — заповнити пробілами).
2. **Намалювати всі ребра + ваги.** Щоб не малювати неорієнтоване ребро двічі (u→v і v→u), ведеться `set<pair<int,int>> drawnEdges` з канонічним ключем `(min, max)`.
3. **Намалювати всі вузли** — поверх ребер, щоб `[id]` не перекривався лінією.

**Чому namespace, а не клас.** У `ConsoleRenderer` немає стану — всі функції чисто беруть параметри і пишуть у консоль. Namespace з inline-функціями семантично точніший, ніж клас зі статичними методами.

## 5. Бібліотека заготовлених топологій

**Файл:** `kursova/TopologyLibrary.h` — namespace `TopologyLibrary`.

**Суть.** Три захардкоджені «гарні» топології, які відрізняються структурою графа, — щоб користувач міг порівнювати алгоритми на якісно різних сценаріях, не витрачаючи час на ручне розміщення.

**Три топології:**

| Функція | Структура | Вузлів | Ребер | Характер |
|---|---|---|---|---|
| `makeStar()` | Зірка з центром `R1` | 6 | 5 | Всі інші з'єднані тільки через центр. Найкоротший шлях від периферії — завжди через `R1`. |
| `makeMesh()` | Щільна сітка | 5 | 8 | Багато альтернативних маршрутів з різною вагою. Тут A\* і Дейкстра можуть розбігатися по кількості розкритих вузлів. |
| `makeLinear()` | Ланцюг `R1–R2–…–R6` | 6 | 5 | Єдиний можливий шлях між кінцями. Гарний для демонстрації того, що всі три алгоритми знаходять однаковий шлях, але з різними метриками. |

**Координати підібрані вручну** так, щоб у полі 80×20 топологія виглядала симетрично і жодне ребро не перекривало вузол. Наприклад, у зірці центр `R1` стоїть у `(40, 10)` — рівно посередині поля, — а периферія розкидана по кутах і бокових серединах.

**Чому вручну, а не генератор.** Згенерована топологія за формулою легко виглядає «погано» (перекручені ребра, мітки накладаються). Для курсового важлива **якість візуального представлення**, а не різноманіття сценаріїв, тому 3 чистенькі варіанти — кращий вибір.

## 6. Ручна побудова графа

**Файл:** `kursova/GraphBuilder.h` — namespace `GraphBuilder`.

**Суть.** Користувач сам будує граф за два кроки: 1) розміщує вузли курсором на полі; 2) вводить ребра текстом.

### 6.1. `placeNodesInteractive()` — розміщення вузлів

**Як це працює:**

1. У полі відображається `+` — позиція курсора. Колір курсора — інверсія (`(10 << 4) | 0` = зелений фон, чорний символ), щоб його було видно на будь-якому тлі.
2. Стрілки рухають курсор (`cursorX`, `cursorY`), з перевіркою меж поля.
3. Enter додає вузол у поточну позицію з автоматичним іменем `R<n>` і наступним `id`.
4. Esc завершує розміщення — після нього граф має якісь вузли, ще без ребер.
5. Після кожної клавіші поле **повністю перемальовується** (`system("cls")` + `drawField`). Це простіше, ніж diff-рендер, і на полі 80×20 працює миттєво.

У нижньому рядку панель стану показує поточні координати курсора, кількість вузлів і наступний `id` — це UX-підказка, щоб користувач не губився.

### 6.2. `addEdgesByText(graph&)` — введення ребер

**Чому текстом, а не кліками на вузли.** У консолі немає миші (принаймні, не без RAWINPUT-ускладнень), а навігація стрілками по всьому полю до кожного вузла була б утомлива. Текст у форматі `<id_from> <id_to> <weight>` — швидкий, явний і не плутає користувача.

**Парсинг:**

```cpp
std::istringstream iss(line);
int u, v;
double w;
if (!(iss >> u >> v >> w)) { /* помилка */ }
```

**Валідація:**

1. Формат (всі три числа прочиталися).
2. Обидва `id` існують у графі.

Якщо щось не так — повідомлення червоним кольором (`setAttr(12)`) і продовження циклу. **Порожній рядок** завершує введення.

### 6.3. `buildInteractive()` — композиція

Об'єднує два кроки в один потік. Якщо після розміщення граф порожній (користувач одразу натиснув Esc), етап ребер пропускається.

**Чому namespace + `detail::` для констант.** Коди клавіш тут ті самі, що в головному меню (`KEY_UP = 72` тощо), але винесено їх у приватний `detail`-namespace, щоб не конфліктувати з `kursova.cpp`, де вони оголошені на верхньому рівні.

## 7. Вибір стартового та цільового вузлів

**Файл:** `kursova/kursova.cpp` — `selectEndpointsAndPreview()` (рядки 52–120).

**Суть.** Після побудови графа користувач вводить два `id` (стартовий і цільовий), перевіряється їх валідність, і вся ця інформація передається у візуалізатор.

**Послідовність:**

1. Гранична перевірка: якщо вузлів менше двох — помилка, повернення.
2. Вивід списку **всіх вузлів** з їхніми координатами, щоб користувач міг обрати осмислено.
3. Введення через `cin >> startId >> targetId` з окремою валідацією кожного (що саме `int` прочитався; що такий `id` існує; що start ≠ target).
4. Підказка клавіш керування анімацією і виклик `AlgorithmVisualizer::runAllAlgorithms(g, start, target)`.

**Чому через `cin`, а не курсор по графу.** Одне числове введення простіше і менш схильне до помилок, ніж «встаньте на вузол X → Enter → встаньте на вузол Y → Enter». Крім того, список вузлів над полем введення і так показує всі `id` — не потрібно шукати їх візуально.

**Обробка помилок `cin`:** після `if (!(cin >> x))` — `cin.clear(); cin.ignore(10000, '\n');`, щоб залишки невалідного вводу не попсували наступні читання.

## 8. Алгоритми маршрутизації

**Файл:** `kursova/RoutingAlgorithms.h` — клас `RoutingAlgorithms` з трьома статичними методами.

**Суть.** Три класичні алгоритми пошуку найкоротшого шляху, з уніфікованою сигнатурою:

```cpp
static RoutingResult algo(const NetworkGraph& g, int start, int target, StepLog* log = nullptr);
```

Уніфікація критична: візуалізатор зберігає масив покажчиків на функції (`AlgoFn`) і викликає їх у циклі, не знаючи, який саме алгоритм працює.

**`RoutingResult`** (рядки 15–22) містить:

- `path` — знайдений шлях як послідовність `id` (або порожній, якщо шляху немає);
- `totalCost` — сумарна вага (або `infinity`);
- `iterations` — скільки ітерацій основного циклу;
- `relaxations` — скільки разів полегшено оцінку відстані (поліпшено `distances[v]`);
- `expandedNodes` — скільки вузлів повністю оброблено;
- `elapsedMicroseconds` — час роботи алгоритму (виміряний через `chrono::high_resolution_clock`).

Усі метрики заповнюються однаково в усіх трьох алгоритмах — це те, що порівняльний звіт потім зіставляє.

### 8.1. Алгоритм Дейкстри

**Метод:** `dijkstra(graph, startId, targetId, log)` (рядки 39–122).

**Ключова структура — min-heap (бінарна купа):**

```cpp
priority_queue<pair<double, int>, vector<pair<double, int>>, greater<...>> pq;
```

Порівняння за `pair.first` (відстань) дає вилучення мінімуму за O(log n).

**Інваріант:** у `settled` лежать вузли з вже фінальною відстанню. Алгоритм зупиняється, тільки коли **ціль витягується з черги** (`if (currentId == targetId) break;`) — це ранній вихід, який дає виграш у швидкості, бо не потрібно обчислювати відстані до всіх інших вузлів.

**Релаксація:**

```cpp
double newDist = currentDist + edge.weight;
if (newDist < distances[edge.to]) {
    distances[edge.to] = newDist;
    previous[edge.to] = currentId;
    result.relaxations++;
    pq.push({ newDist, edge.to });
}
```

Тобто в чергу можуть потрапити **дублікати** з різними відстанями — але оскільки ми беремо мінімум і перевіряємо `currentDist <= distances[currentId]`, застарілі записи просто пропускаються. Це «lazy deletion» — простіше, ніж реалізовувати decrease-key.

**Відновлення шляху** йде від цілі назад за `previous[]` і потім реверсується — класика.

### 8.2. Алгоритм Беллмана-Форда

**Метод:** `bellmanFord(graph, startId, targetId, log)` (рядки 124–217).

**Ключова відмінність від Дейкстри:** не використовує пріоритетну чергу; натомість виконує **`|V|-1` проходів**, і на кожному проході ітерує по **всіх ребрах**, намагаючись зробити релаксацію:

```cpp
for (int i = 0; i < vertexCount - 1; ++i) {
    for (each node u) {
        for (each edge u -> v) {
            if (distances[u] + w < distances[v]) { relax(); }
        }
    }
}
```

**Оптимізація — ранній вихід:** якщо за поточний прохід **жодної релаксації** не сталося (`!anyRelaxation`), немає сенсу робити наступні — алгоритм зупиняється.

**Перевірка від'ємних циклів:** після основного циклу виконується ще один повний прохід. Якщо він ЩЕ раз покращує хоч якусь відстань — граф містить від'ємний цикл, досяжний зі старту; алгоритм чесно повертає `infinity`. Для поточного застосунку (ваги = затримки, завжди ≥ 0) цей випадок не спрацьовує, але для коректності алгоритму він є.

**Чому Беллман-Форд, якщо ваги ≥ 0?** Бо він передбачений формулюванням курсової та є класичним «антиподом» Дейкстри — різна асимптотика (`O(VE)` проти `O((V+E) log V)`) створює контраст у звіті. На звичайній топології ітерацій у Беллман-Форда суттєво більше.

### 8.3. Алгоритм A*

**Метод:** `aStar(graph, startId, targetId, log)` (рядки 219–308).

**Ідея.** Та сама пріоритетна черга, що й у Дейкстри, але пріоритет = `f = g + h`, де:

- `g[v]` — реальна найкраща відома вартість від старту до `v`;
- `h(v)` — **евристична** оцінка відстані від `v` до цілі.

**Евристика:** евклідова відстань за координатами `Node.x, Node.y`:

```cpp
static double euclideanHeuristic(const Node& a, const Node& b) {
    double dx = a.x - b.x, dy = a.y - b.y;
    return sqrt(dx * dx + dy * dy);
}
```

**Чому саме евклідова.** Вона **допустима** (ніколи не переоцінює реальну відстань), якщо ваги ребер ≥ евклідова довжина ребра. Для заготовлених топологій це так (ваги — цілі, координати — в піксельних одиницях, що різняться за шкалою). Тобто A\* гарантовано знайде оптимальний шлях, але розкриє менше вузлів, ніж Дейкстра — бо евристика «тягне» пошук у напрямку цілі.

**Closed list (`closed`):** коли вузол розкритий, він сюди додається і більше ніколи не перевідкривається. Якщо сусід уже в `closed`, його пропускаємо. Це коректно для допустимих і консистентних евристик.

**Чому той самий `priority_queue`, не `set`.** Щоб не писати decrease-key — просто пушимо нову пару `(f, id)` і ігноруємо застарілі через перевірку `closed.find(currentId)`.

**Чому один клас на всі три алгоритми.** Вони ділять структуру `RoutingResult`, однакову сигнатуру і схожі допоміжні структури (`distances`, `previous`). Об'єднання в один клас спрощує імпорт для візуалізатора.

## 9. Покрокове журналювання (StepLog)

**Файл:** `kursova/RoutingAlgorithms.h` — структура `StepEvent` і тип `StepLog = vector<StepEvent>` (рядки 24–35).

**Суть.** Кожен алгоритм **під час своєї роботи** додає в опційний `StepLog*` знімок свого стану на кожному значущому кроці. Візуалізатор потім програє цей журнал.

**Що в знімку:**

- `currentId` — який вузол обробляється зараз;
- `currentDist` — його відстань у цю мить;
- `visited` — усі вже остаточно оброблені вузли;
- `frontier` — вузли, до яких уже є оцінка, але ще не фінальна;
- `iterations`, `relaxations`, `expandedNodes` — інкрементальні лічильники;
- `note` — текстовий коментар (Беллман-Форд використовує для «Прохід N/M»).

**Чому так, а не callback:**

1. **Відокремлення обчислення від рендеру.** Алгоритм тупо заповнює дані — він нічого не знає про консоль, розмір поля, кольори чи паузу. Це спрощує тестування (журнал можна роздрукувати в текст) і переносимість.
2. **Повторне відтворення.** Можна промотати журнал вперед/назад без повторного запуску алгоритму.
3. **Опційність.** `log = nullptr` за замовчуванням → працюючий алгоритм без журналу не витрачає пам'ять.

**Як це використовується:**

```cpp
StepLog log;
RoutingResult res = fn(g, startId, targetId, &log);   // заповнює log
for (const auto& ev : log) { repaintStep(...); sleep(...); }  // програємо
```

## 11. Анімація передачі пакета

**Файл:** `kursova/AlgorithmVisualizer.h` — `animatePacket()` (рядки 206–252).

**Суть.** Після того як алгоритм знайшов шлях, вздовж кожного ребра шляху по черзі «пролітає» символ `*`.

**Як це працює:**

1. Для кожної пари сусідніх вузлів у шляху:
   - Обчислюється кількість кроків: `steps = max(|dx|, |dy|)`.
   - Обирається символ ребра (`edgeCharFor` — той самий, що в `drawLine`).
   - На кожному кроці `step = 1..steps-1`:
     - Попередня позиція пакета переписується символом ребра (`\\`, `-` тощо) — тобто за пакетом залишається підсвічений слід маршруту.
     - На новій позиції малюється `*` жовтим.
     - `sleepResponsive(d, ss, ...)` з затримкою `max(30, delay/3)` — пакет рухається **утричі швидше**, ніж покрокова візуалізація, щоб не засипати глядача.
2. Дійшовши до чергового вузла, малюється повноцінний `[id]` кольором `COLOR_NODE_PATH` (або `COLOR_NODE_TARGET` для останнього).

**Чому «за пакетом залишається слід».** Це робить помітним факт, що пакет реально проходить саме тим ребром, а не телепортується. Без цього було б видно тільки один стрибаючий символ.

**Чому швидкість пакета не дорівнює delay.** Кроки пакета — це «піксельні» дрібніші одиниці (кожен символ вздовж ребра), а кроки алгоритму — це «дискретні» події в логіці. На швидкості 3 (220 мс) пакет рухався б по 220 мс на пікель → ребро їхав би секунди. Тому для пакета використовується `delay / 3` з мінімумом 30 мс.

## 12. Керування швидкістю, пауза, пропуск

**Файл:** `kursova/AlgorithmVisualizer.h` — `SpeedState` (рядки 37–41), `pollInput()` (108–125), `sleepResponsive()` (127–150), `delayMsForLevel()` (43–51).

**Суть.** Користувач може в реальному часі змінювати швидкість, ставити на паузу або пропустити поточний алгоритм — не чекаючи кінця анімації.

**`SpeedState`:**

```cpp
struct SpeedState {
    int level = 3;          // 1..5
    bool paused = false;
    bool skip = false;
};
```

**Таблиця затримок:**

| Рівень | Затримка (мс) |
|---|---|
| 1 | 900 |
| 2 | 450 |
| 3 | 220 (за замовчуванням) |
| 4 | 110 |
| 5 | 55 |

Геометрична прогресія з кратністю ~2 — тобто сусідні рівні відчутно відрізняються.

**`pollInput()` — неблокуюче читання клавіатури:**

```cpp
while (_kbhit()) {
    int key = _getch();
    if (key == 224) {
        int k2 = _getch();
        if (k2 == 72) { if (ss.level < 5) ss.level++; }       // ↑
        else if (k2 == 80) { if (ss.level > 1) ss.level--; }  // ↓
    } else if (key == ' ') { ss.paused = !ss.paused; }
    else if (key == 27) { ss.skip = true; }                    // Esc
    else if (key == '+' || key == '=') { ... }
    else if (key == '-' || key == '_') { ... }
}
```

`_kbhit()` — чи є щось у буфері. Поки є — читаємо, але не чекаємо. Це дозволяє буквально в циклі анімації реагувати на клавіші без зависань.

Символи `+`/`-` підтримуються як запасний варіант, бо залежно від розкладки стрілки можуть по-різному оброблятися.

**`sleepResponsive(ms, ss, ...)` — «розумна затримка»:**

```cpp
auto start = steady_clock::now();
while (true) {
    pollInput(ss);
    if (ss.skip) return;
    if (ss.paused) { Sleep(30); continue; }
    if (elapsed >= ms) return;
    Sleep(15);
}
```

- Блок `while (ss.paused)` — ефективна пауза: ми не рухаємося далі, але продовжуємо перевіряти ввід кожні 30 мс (щоб миттєво зняти паузу на Space).
- Якщо за час очікування змінився рівень або стан паузи — перемальовується інформаційна панель (`drawInfoPanel`). Це реактивний UX: користувач бачить нове значення одразу.
- `Sleep(15)` — opt. частота опитування клавіатури ≈60 разів/с, що вище за час реакції людини.

**`skip`** — односторонній прапор «перескочити поточний алгоритм»: `runSingle` його перевіряє після кожного кроку і між етапами (покрокова візуалізація → анімація пакета), і при true обриває цикл, перемальовує тільки фінальний шлях і повертає результат.

## 14. Порівняльний звіт

**Файл:** `kursova/AlgorithmVisualizer.h` — `renderComparisonReport()` (рядки 335–514), `MetricRow` (309–314).

**Суть.** Фінальний екран, який зіставляє три алгоритми за набором метрик у вигляді ASCII-таблиці, підсвічує найкращі/найгірші значення та виводить одне-реченнєвий висновок.

**Метрики:**

- **Довжина** — кількість ребер у шляху (`path.size() - 1`);
- **Вартість** — сумарна вага шляху (`totalCost`);
- **Ітерацій** — кроків основного циклу алгоритму;
- **Релаксацій** — скільки разів покращено оцінку відстані;
- **Розкрито** — скільки вузлів оброблено;
- **Час, мкс** — реальний час роботи.

**Структура `MetricRow`:**

```cpp
struct MetricRow {
    std::string label;
    std::string values[3];      // відформатовані для виводу
    bool isValid[3];            // чи є шлях у цього алгоритму
    double numerics[3];         // значення для порівняння
};
```

Дві версії заповнювачів — `fillIntRow` (цілі — кількості та час) і `fillDoubleRow` (вартість з одним знаком після коми), щоб формат був осмислений.

**Побудова таблиці:**

1. Ширина колонки лейбла = `max(|"Метрика"|, всіх лейблів рядків)`.
2. Ширина кожної з трьох колонок значень = `max(|algoName|, всіх значень у колонці)`.
3. `padRightStr` доповнює пробілами справа до потрібної ширини.
4. Роздільники — `+---+---+---+---+` з `printSeparator`.

**Кольорове виділення:**

- Для кожного рядка шукаються `bestVal` і `worstVal` серед **валідних** значень.
- Якщо всі рівні — без кольору.
- Інакше: значення, що дорівнює `bestVal` — зелене (10), `worstVal` — червоне (12).

**Підрахунок «переможця за метрикою»:** якщо `winners.size() == 1`, цей алгоритм отримує `+1` до `wins[i]`. Важливо: **при нічиях ніхто бали не отримує** (вибіркова справедливість — не штучно визначати лідера з кількох рівних).

**Висновок:** «найефективнішим на цьому графі виявився X (лідер за N/M метриками)». `topIdx = argmax(wins)`, `total = rows.size()`.

**Чому не зафіксовано єдиного критерію «переможця».** Кожна метрика показує свій аспект (якість шляху vs. ресурси, витрачені на пошук), тому найчесніше — голосування, а не зважена сума, яка вимагала б штучних коефіцієнтів.

**Секція «Шляхи»** унизу — просто перелік знайдених шляхів кожного алгоритму у вигляді `R1 -> R3 -> R5`, або «не знайдено» сірим.

## Як усе зв'язується разом

Типовий сценарій «користувач запустив — побачив звіт»:

1. **main** → встановлює CP1251, ховає курсор, рендерить головне меню.
2. Вибір **Play** → `showPlay` → вибір способу побудови графа.

4. `finalizeGraph` → `renderGraphScreen` (показ готового графа) + `selectEndpointsAndPreview` (вибір start/target).
5. `AlgorithmVisualizer::runAllAlgorithms`:
   - для кожного з 3 алгоритмів: `RoutingAlgorithms::<fn>(g, start, target, &log)` → заповнений `StepLog` і `RoutingResult`;
   - `runSingle` програє лог крок за кроком (`repaintStep`), далі підсвічує шлях, далі анімує пакет.
6. `renderComparisonReport` — таблиця, підсвічені найкращі/найгірші значення, висновок, шляхи.
7. Повернення у меню.

Увесь граф даних проходить через **один тип** — `NetworkGraph`; усі алгоритми мають **одну сигнатуру**; увесь рендер іде через **одну область 80×20**. Саме ця уніфікація робить проєкт акуратним: додавання четвертого алгоритму = один новий статичний метод у `RoutingAlgorithms` + один рядок у масиві `algos[]`.
